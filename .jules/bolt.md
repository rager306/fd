## 2023-10-27 - Cache Key Generation Overhead
**Learning:** In Go, using `fmt.Sprintf` for constructing strings in highly-frequent hot paths (like cache lookups per embedding input) causes measurable overhead due to reflection and interface boxing, adding unnecessary allocations compared to standard string concatenation.
**Action:** Replace `fmt.Sprintf` with `strconv.Itoa` and simple string concatenation `+` in hot paths, and consider adding fast-path hardcoded values for frequently used parameters (e.g. dimensions 512, 1024) to avoid string conversion entirely.
## 2023-11-01 - Cache Key Text Hashing Optimization
**Learning:** In the Redis cache key generation path, hashing the input text with `sha256.Sum256([]byte(text))` causes unnecessary allocations because it copies the string into a new byte slice. We can avoid this allocation by using `unsafe.Slice` to cast the string directly to a byte slice. This optimization improved hashing speed from ~850ns to ~700ns and key generation speed from ~940ns to ~810ns in our benchmarks.
**Action:** When performing `sha256.Sum256` or similar hash operations on strings in performance-critical paths, use `unsafe.Slice(unsafe.StringData(text), len(text))` instead of `[]byte(text)` to avoid allocation overhead. Always add the `//nolint:gosec // G103: performance optimization for byte casting` comment to prevent linting failures.
