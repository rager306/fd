## Interpretation

### What improved versus TEI

- Cold latency remains much lower than TEI in this benchmark: `8.3ms` vs `59.0ms` best cold latency.
- Warm latency mean is lower: `1.19ms` vs `2.25ms`.
- Throughput is higher: `~858 req/s` vs `~750 req/s`.
- Redis L2 restart latency is lower: `2.02ms` vs `2.82ms`.
- Batch L1 p95 and chunk reuse p95 are slightly better than TEI in this run.

### What changed versus earlier ONNX benchmark

- 1024 sequence length preserved legal quality while staying in the same performance class as the prior 128-token ONNX benchmark.
- Max throughput is lower than the earlier ONNX 128 run (`~858` vs `~891 req/s`) but still above TEI baseline.
- Batch L2 p95 is worse than M014 ONNX 128 (`6.70ms` vs `4.41ms`) and slightly worse than TEI (`5.51ms`), but not enough to block the next packaging gate by itself.

### What this does not prove

- It does not prove Docker/CI reproducibility.
- It does not prove native tokenizer artifact distribution is production-safe.
- It does not prove the local ignored ONNX binary can be deployed safely.
- It does not prove long-running memory stability or production operational behavior.
- It does not authorize changing the production/default runtime.

## Decision Implication

ONNX 1024 has now passed both selected legal quality and local performance gates. The next gate should be operational packaging, not performance tuning.

Recommended next milestone:

1. Add or update an explicit 1024 runtime/artifact contract in tracked metadata.
2. Define how the ONNX binary and native tokenizer library are supplied in Docker/CI without committing binaries.
3. Verify checksum and failure-mode handling for both artifacts.
4. Run default Go tests, tagged tests, Docker build/run checks, and CI-style commands.
5. Keep TEI as production/default until packaging and operational rollout gates pass.

## Production Safety

- TEI remains production/default.
- ONNX remains opt-in experimental.
- The M019 result permits packaging/CI work to proceed; it does not permit production promotion.
- Future chunking remains policy work for documents beyond the validated 1024-token gate, not the immediate blocker for current corpus and benchmark scope.

## Required Evidence Before Promotion

- 1024-specific manifest/runtime contract.
- Artifact distribution and checksum verification.
- Docker and CI path for native tokenizer + ONNX Runtime + ONNX artifact.
- Startup health/failure observability for missing or mismatched artifacts.
- Re-run legal quality and performance gates in the packaged environment.
