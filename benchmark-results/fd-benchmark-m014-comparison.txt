# fd M014 Benchmark Comparison — TEI default vs tagged ONNX HF tokenizer

## Inputs

- TEI baseline: `benchmark-results/fd-benchmark-m014-tei-baseline.txt`
- Tagged ONNX: `benchmark-results/fd-benchmark-m014-onnx-hf-tokenizer.txt`
- Correctness gate: `benchmark-results/fd-go-onnx-hf-tokenizer-m013-s03.txt` (`PASS`, cosine threshold `0.999`)
- Raw benchmark/probe texts are intentionally excluded from artifacts.

## Runtime modes

- TEI: default Compose API on `http://localhost:8000`, snapshot v2, production/default runtime.
- Tagged ONNX: local opt-in `hf_tokenizers` build on `http://localhost:18000`, snapshot v3, isolated Redis namespace `m014-onnx-hf-tokenizer`.
- Tagged ONNX artifact metadata records native tokenizer, ONNX model, and ONNX Runtime library checksums.

## Interpretation

Tagged ONNX is materially faster on the uncached/cold model-bound path in this local benchmark. The largest gains are best cold latency (`5.78x` faster) and cold batch prime time (TEI `3144.81ms` vs ONNX `604.64ms`, about `5.20x` faster from the source artifacts). Warm/cache-dominated paths are closer because Redis and in-process cache dominate once vectors are cached.

Throughput peak is slightly higher for ONNX in this run (`+18.8%`), but the peak occurs at 4-way concurrency for ONNX vs 16-way for TEI, so concurrency tuning and repeated runs are needed before treating this as a stable capacity result.

## Caveats

- This is a fixed local host benchmark on the captured KVM/QEMU AMD EPYC environment, not production hardware.
- ONNX correctness is based on fixed-probe cosine equivalence from M013, not a broad Russian/legal retrieval evaluation.
- Tagged ONNX still depends on local ignored artifacts: `.gsd/runtime/onnx/...` and `.gsd/runtime/tokenizers/...`.
- Native tokenizer packaging is not Docker/CI-ready and the native manifest still references an upstream `latest` release URL.
- Startup/RSS is operationally significant: S03 observed about 1.69 GiB RSS shortly after tagged ONNX startup.
- TEI remains the production/default backend. No default switch is made by M014.

## Recommendation

Continue ONNX work, but only as an opt-in experimental path. The benchmark evidence justifies a follow-up packaging/tuning milestone for Docker/CI-native artifacts and repeated benchmark runs. It does not justify switching production default away from TEI yet.
