# S03: ONNX conditional fallback and speed measurement

**Goal:** Opt-in ONNX mode and speed measurement with M043 gates and known ONNX/cache gotchas built into the plan: safe env/operator path handling, tokenizer parity awareness, cache namespace isolation, govulncheck, and 18-linter fail mode.
**Demo:** After this, FD_BACKEND=onnx (requires rebuilding fd binary с -tags onnx) переключает fd на Go ONNX runtime. Per M019: cold path batch=32 ≤500ms, warm path ≤10ms. Default off (TEI остаётся production). Legal quality gate deferred per M015/M016.

## Must-Haves

- ONNX binary builds clean with `go build -tags onnx`.
- FD_BACKEND=onnx cold path batch=32 <=500ms and warm path batch=1 <=10ms, or documented blocker with evidence.
- Default FD_BACKEND=tei remains unchanged.
- M041 acceptance tests and M042 async regression tests pass in ONNX mode where applicable.
- `go test ./...`, 18-linter golangci-lint, and govulncheck all exit 0 after ONNX changes.
- Cache namespace is isolated between TEI and ONNX runs to avoid false equivalence.

## Proof Level

- This slice proves: Build proof + runtime perf proof + regression proof + static-analysis/security proof.

## Integration Closure

ONNX remains opt-in, TEI remains production default. Legal quality gate remains deferred unless separately scoped; docs must state runtime identity and cache namespace requirements.
