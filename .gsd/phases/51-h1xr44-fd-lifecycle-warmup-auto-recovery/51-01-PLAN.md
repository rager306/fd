# S01: Warmup recovery contract

**Goal:** fd-api восстанавливает readiness автоматически после провала стартового warmup через periodic background retry, пока TEI достижим и процесс не в shutdown. Стартовое окно расширено до ~30с. Поведение конфигурируемо через env, по умолчанию включено, откатываемо.
**Demo:** После `docker compose down && up` с задержанным TEI fd-api приходит в `/health` 200 без ручного restart

## Must-Haves

- 1. После провала стартового warmup fd-api продолжает попытки в фоне, пока state.IsWarmupDone() не станет true или процесс не начнёт shutdown.
- 2. Стартовое окно: ≥5 попыток с backoff ≥5с (env: FD_WARMUP_START_MAX_ATTEMPTS, FD_WARMUP_START_BACKOFF_SEC).
- 3. Recovery: periodic retry с интервалом FD_WARMUP_RECOVERY_INTERVAL_SEC (default 30), feature flag FD_WARMUP_RECOVERY_ENABLED (default true).
- 4. docker compose down && up с TEI, стартующим за 30-60с, даёт /health 200 warmup_done:true без ручного restart fd_api.
- 5. Recovery останавливается при shutdown и при успешном warmup.
- 6. Логи: каждая recovery-попытка структурированно логируется; /health.last_error отражает актуальную причину.
- 7. go test ./api/... ./api/lifecycle/... проходит, новые тесты покрывают: recovery retry до успеха, остановку при shutdown, остановку при успехе, feature flag off.
- 8. Существующие TestStartModelWarmup* не сломаны (или осмысленно обновлены).
- 9. /health shape и middleware.LifecycleGate не изменены.
